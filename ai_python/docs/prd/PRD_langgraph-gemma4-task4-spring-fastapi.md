## 4.1. Project Overview

Core goal: define stable integration touchpoints between upstream Spring gateway and `ai_python` FastAPI for LangGraph execution, including invoke/stream contracts, required metadata propagation, correlation tracing, error envelope, and security boundaries. This PRD only defines Python-side API contracts and integration behavior; it does not implement Spring/backend/frontend code.

Target users/actors:
- Upstream Spring service (service-to-service caller)
- FastAPI service in `ai_python` (LangGraph runtime)
- Observability/ops maintainers (trace, logs, incident triage)

### Goals
- Standardize 2 API modes: sync invoke and streaming.
- Enforce metadata contract: `user_id`, `tenant_id`, `thread_id`, `schema_version`.
- Preserve end-to-end tracing with `X-Correlation-Id`.
- Keep provider credentials server-side; never expose LLM API key to clients.

### Non-goals
- No Spring controller implementation details.
- No frontend contract changes.
- No redesign of Task 1/2/3 internals (LLM port, graph topology, agent behavior).
- SQL execution posture for production is **via Spring proxy** (`http_spring`); Python-only DB paths are dev/CI only unless explicitly allowed by env (per Option B).

## 4.2. Specifications

### Functional requirements

#### FR-01: FastAPI endpoints (touchpoints)
- `POST /api/v1/ai/chat/invoke`: synchronous response, final answer envelope.
- `POST /api/v1/ai/chat/stream`: streaming response over **SSE** (`text/event-stream`).

#### FR-02: Required headers and metadata
- Required request header: `X-Correlation-Id` (UUID/string, generated by Spring if absent from upstream client).
- Required body metadata:
  - `user_id` (string, non-empty)
  - `tenant_id` (string, non-empty)
  - `thread_id` (string, optional but recommended for checkpoint continuity)
  - `schema_version` (string, default fallback configured in Python if omitted)

#### FR-03: Request shape (canonical)
```json
{
  "message": "Doanh thu thang nay cua chi nhanh HN la bao nhieu?",
  "metadata": {
    "user_id": "u-123",
    "tenant_id": "tenant-a",
    "thread_id": "t-001",
    "schema_version": "v1"
  },
  "options": {
    "stream": false,
    "locale": "vi-VN"
  }
}
```

#### FR-04: Invoke response shape
```json
{
  "correlation_id": "0f8fad5b-d9cb-469f-a165-70867728950e",
  "thread_id": "t-001",
  "intent": "system_data_query",
  "final_answer": "Doanh thu thang nay cua chi nhanh HN la 1.2 ty VND.",
  "usage": {
    "input_tokens": 0,
    "output_tokens": 0
  },
  "error": null
}
```

#### FR-05: Error envelope (invoke + stream terminal error)
```json
{
  "correlation_id": "0f8fad5b-d9cb-469f-a165-70867728950e",
  "error": {
    "code": "AI_VALIDATION_FAILED",
    "message": "SQL validation failed after max attempts",
    "details": {
      "attempts": 3
    }
  }
}
```

#### FR-06: Streaming event contract
- Event payload minimum fields: `correlation_id`, `event_type`, `delta` or `data`, `is_terminal`.
- Event types: `token`, `partial_answer`, `final_answer`, `error`, `heartbeat` (optional).
- Stream must terminate with one terminal event (`final_answer` or `error`).

#### FR-07: Metadata forwarding to graph runtime
- FastAPI must pass metadata into graph config/state (compatible with Task 2 assumptions and Task 3 agents):
  - `correlation_id` → logging and trace context
  - `tenant_id` / `user_id` → policy and audit context
  - `thread_id` → checkpointer continuity
  - `schema_version` → `SchemaLoader` selector

#### FR-08: Compatibility with Task 1–3
- Task 1: use existing `LlmClient` abstraction; no direct provider object creation in API layer.
- Task 2: use existing graph compile, retry cap, checkpointer, and stream hooks.
- Task 3 agents: preserve agent node behavior and feedback contract; API layer only orchestrates IO.

### Locked architecture — **Option B** (Owner selection)

- **Auth / context:** Spring validates end-user auth; service calls FastAPI with **JWT (or signed context)** whose claims FastAPI validates (issuer/audience/signature via JWKS or shared verification contract). Claims MUST include at least `user_id`, `tenant_id`; MAY include roles/scopes used for graph policy hints.
- **Stream transport:** **SSE** only (`text/event-stream`) for `POST /api/v1/ai/chat/stream`.
- **SQL execution posture:** **Production** path uses **Spring-backed HTTP proxy** (`http_spring`) for read-only query execution with centralized authorization; stub/python read-only modes remain acceptable for **non-prod** environments only.
- **Rationale:** Aligns tenant and data-access governance with the ERP/Spring layer while keeping invoke/stream contracts stable in `ai_python`.

**Alternatives not selected:** Option A (shared-secret m2m, flexible Python SQL), Option C (mTLS + NDJSON + hybrid executor) — documented for history only; do not implement unless a new PRD revision is opened.

### Non-functional requirements (quantified)
- NFR-PERF-01: `POST /invoke` p95 latency < 8s for standard text prompt (excluding external outages).
- NFR-PERF-02: stream first token/event p95 < 3s in normal provider conditions.
- NFR-REL-01: API 5xx rate < 1% on steady-state traffic.
- NFR-REL-02: terminal event guarantee for 100% stream sessions (`final_answer` or `error`).
- NFR-SEC-01: 0 secrets in responses/logs (`LLM_API_KEY`, raw credential headers).
- NFR-OBS-01: 100% requests include propagated `X-Correlation-Id` in logs.
- NFR-COMP-01: backward compatibility for request/response schema via `schema_version`.

### Security requirements
- Never expose provider `API_KEY` to client-facing layers or logs.
- Under **Option B**, inbound requests MUST carry a **valid JWT** (or equivalent signed context) from Spring; FastAPI validates signature/issuer/audience and derives `user_id` / `tenant_id` from claims (and rejects on failure).
- FastAPI trusts identity/context only from upstream Spring/service caller boundary.
- Reject missing/invalid metadata (`user_id`, `tenant_id`) with 4xx error.
- Mask sensitive SQL/content fields in logs according to Task 2 logging policy.

## 4.4. Task breakdown & dependency graph

- [ ] **Task 4.1 — Define API schema module**
  - **Description:** Freeze request/response/error envelope models for invoke/stream touchpoints, including metadata and `X-Correlation-Id`.
  - **Input/Output:** Input = this PRD + Task 1–3 contracts; Output = typed API schemas in `ai_python`.
  - **Acceptance Criteria:** Both endpoints validate schema; invalid metadata returns 4xx with standard error envelope.

- [ ] **Task 4.2 — Implement invoke endpoint integration**
  - **Description:** Map request→graph invocation→canonical response envelope; propagate correlation and metadata.
  - **Input/Output:** Input = API request, compiled graph callable; Output = invoke JSON response.
  - **Acceptance Criteria:** Returns `final_answer` on success, standardized `error` envelope on failure, preserves `thread_id`.

- [ ] **Task 4.3 — Implement stream endpoint integration**
  - **Description:** Expose stream API over **SSE**, terminal event guarantee, error mapping.
  - **Input/Output:** Input = API request with stream option; Output = streaming events.
  - **Acceptance Criteria:** Emits valid event sequence and exactly one terminal event.

- [ ] **Task 4.4 — Correlation and audit propagation**
  - **Description:** Ensure `X-Correlation-Id` and metadata are attached to graph context and logs consistently.
  - **Input/Output:** Input = headers + metadata; Output = structured logs/trace fields.
  - **Acceptance Criteria:** Log sampling shows 100% correlation presence for successful and failed requests.

- [ ] **Task 4.5 — Security guardrails**
  - **Description:** Enforce trusted caller/auth policy per selected option; redact secrets and sensitive fields.
  - **Input/Output:** Input = inbound auth/context; Output = accepted/rejected requests and safe logs.
  - **Acceptance Criteria:** No API key leakage in responses/logs; unauthorized requests blocked with 401/403.

- [ ] **Task 4.6 — Contract tests for touchpoints**
  - **Description:** Add integration-style tests for invoke/stream envelopes, metadata validation, and error mapping.
  - **Input/Output:** Input = test fixtures and mock graph/LLM responses; Output = passing contract test suite.
  - **Acceptance Criteria:** Tests cover success/fail paths for both endpoints and metadata-required constraints.
